# i18n — App Language Support

**Date:** 2026-04-29
**Status:** Approved

## Summary

Add internationalization (i18n) to tulia.study. Ship with Spanish and English. Language preference is auto-detected from the OS locale and can be changed in Settings. All visible UI strings are translated.

---

## Scope

- **Languages v1:** Spanish (`es`) and English (`en`)
- **Strings covered:** All visible UI — settings, nav panels, auth modal, commentary panel, notes, search, toasts, command palette, keyboard shortcuts panel, empty states, tooltips, context menu, breadcrumb bar, reading toolbar
- **Out of scope:** Bible text and commentary content (already language-specific via Bible version selection)

---

## Architecture

### Library

`react-i18next` + `i18next`. Initialized as a singleton in `src/lib/i18n.ts`. No async loading — both locale files are statically imported so there is no render flicker or loading state.

### Locale Files

```
src/locales/
  en.json   ← source of truth for TypeScript types
  es.json   ← same keys, Spanish values
```

Flat key structure, dot-separated by section (`settings.theme`, `notes.placeholder`, etc.). No namespaces — the string count is small enough that splitting adds no benefit.

### TypeScript Integration

`i18next` module augmentation points at `en.json` as the reference type:

```ts
// src/types/i18next.d.ts
import en from '../locales/en.json'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation'
    resources: { translation: typeof en }
  }
}
```

This gives compile-time key checking and autocompletion for all `t()` calls. Missing keys in `es.json` fall back to `en` at runtime — no crash.

### Initialization (`src/lib/i18n.ts`)

```ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import en from '@/locales/en.json'
import es from '@/locales/es.json'

const savedLocale = localStorage.getItem('locale')
const osLocale   = navigator.language.startsWith('es') ? 'es' : 'en'

i18n.use(initReactI18next).init({
  lng: savedLocale ?? osLocale,
  fallbackLng: 'en',
  resources: {
    en: { translation: en },
    es: { translation: es },
  },
  interpolation: { escapeValue: false },
})

export default i18n
```

Imported once at the top of `src/main.tsx` before `ReactDOM.createRoot`.

## State Management

`useUIStore` gains:

```ts
locale: 'en' | 'es'
setLocale: (l: 'en' | 'es') => void
```

`setLocale` writes to `localStorage`, calls `i18n.changeLanguage(l)`, and updates store state. On store init, `locale` is read from `localStorage` — the same pattern used by `theme`, `fontSize`, and `readingMode`.

The store does **not** own the i18n singleton — it only reflects the current preference and triggers changes. Components read language state via `useTranslation()` (reactive to `i18n.changeLanguage()`), not from the store.

## Settings UI Change

A **Language** row is added to the `Appearance` section in `SettingsModal`, between Theme and Font size:

```
Appearance
  Theme      [ 🌙 Dark ] [ ☀️ Light ]
  Language   [ ES ] [ EN ]
  Font size  [ S ] [ M ] [ L ]
```

The buttons use the same segmented-button style as the Theme toggle. Switching language takes effect immediately — the modal re-renders in the new language on the next tick via `useTranslation()` reactivity.
